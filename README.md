# WireTransmit

**通过 3.5mm 音频线在计算机之间传输文件的 Python 工具。**

基于软件 FSK/4FSK 调制解调器实现，支持双向握手、Reed-Solomon 前向纠错和选择性重传，在普通音频线上实现可靠数据传输。

> 
> ⚡ 本项目由 **AI 辅助创作**，基于 Python 工作流在 CodeBuddy 环境中协作完成。

## 快速开始

### 硬件连接

用一根 **3.5mm 公对公音频线** 将两台电脑连接起来：

```
[发送端电脑]  ─── 扬声器/耳机口 ──╮
                                  ├── 3.5mm 音频线 ──┐
[接收端电脑]  ─── 麦克风/线路输入口 ─╯                 │
                                                       │
                （双向通信需要两端都有扬声器+麦克风） ←──╯
```

### 安装

```bash
# 直接克隆或下载项目
cd wiretransmit
pip install -r requirements.txt
```

### 基本使用

**务必先启动接收端，再启动发送端。**

```bash
# 接收端（先启动）
python receive.py received_file.png

# 发送端（后启动）
python send.py original_file.png
```

传输过程会显示四阶段进度：

```
[1/4] 握手 ...
  收到 ACK — 接收端已就绪。

[2/4] 数据传输 ...
  发送 [██████████████████████████████] 100.0%  (8/8)
  所有数据包已发送。

[3/4] 校验 ...
  未收到 NACK — 假设所有包均已成功接收。

[4/4] 完成确认 ...
  收到 DONE — 传输完成。
```

### 查看音频设备

```bash
python send.py --list-devices
```

## 使用指南

### 常用命令行示例

```bash
# 追求速度：提高波特率到 600
python send.py large_file.bin --baud 600
python receive.py output.bin --baud 600

# 信道质量差：降低波特率换取稳定性
python send.py important.dat --baud 100
python receive.py output.dat --baud 100

# 切换回经典 2-FSK（兼容性更好）
python send.py data.bin --mode 2fsk
python receive.py output.bin --mode 2fsk

# 指定音频设备（多声卡或多接口时）
python send.py file.bin --device "扬声器"
python receive.py out.bin --device "麦克风"

# 更长的监听超时（大文件或低速传输时）
python receive.py out.bin --timeout 300

# 播放测试音验证音频输出
python send.py --test-tone
```

### 波特率选型建议

| 使用场景 | 推荐波特率 | 调制方式 | 净速率 | 1 MB 耗时 |
|----------|-----------|---------|--------|----------|
| 远距离 / 老设备 / 噪声环境 | 100 | 4-FSK | ~170 bps | ~48 秒 |
| **日常使用（默认）** | **300** | **4-FSK** | **~500 bps** | **~17 秒** |
| 短距离 / 新设备 / 优质线材 | 600 | 4-FSK | ~1000 bps | ~8 秒 |
| 极限兼容性（降级模式） | 100 | 2-FSK | ~85 bps | ~95 秒 |

> 发送端和接收端的 `--baud` 和 `--mode` 参数必须保持一致。

## 系统架构

```
┌─────────────────────────────────────────────────────────────┐
│                      发送端 (Transmitter)                    │
│                                                             │
│  文件 → RS编码 → 分包 → 4-FSK调制 → 音频输出 (扬声器)        │
│                                         │                   │
│  音频输入 (麦克风) ← 4-FSK解调 ←协议解析 ←┼── ACK/NACK/DONE  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                          │ 3.5mm 音频线
                          │
┌─────────────────────────────────────────────────────────────┐
│                      接收端 (Receiver)                      │
│                                                             │
│  音频输入 (麦克风) → 同步检测 → 4-FSK解调 → 拼帧 → RS解码   │
│                                                             │
│  音频输出 (扬声器) ← 4-FSK调制 ←── ACK/NACK/DONE ──→ 文件   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 模块说明

| 模块 | 职责 |
|------|------|
| `constants.py` | 物理层和协议层参数集中管理 |
| `modulator.py` | 相位连续 FSK/4-FSK 调制，升余弦包络整形 |
| `demodulator.py` | 非相干正交匹配滤波解调 + 同步检测 |
| `framing.py` | 帧构建/解析、CRC-16/CRC-32 校验 |
| `ecc.py` | RS(255,239) 前向纠错编解码器（纯 Python） |
| `protocol.py` | 双向数据包协议（HELLO/ACK/DATA/NACK/DONE） |
| `transmitter.py` | 发送端状态机（握手→传输→重传→完成） |
| `receiver.py` | 接收端状态机（监听→收集→NACK→校验→保存） |
| `utils.py` | 设备列表、进度显示、SHA-256 校验 |

## 协议设计

### 数据包格式

```
┌──────────┬────────┬──────────────────┬────────────┬──────────┐
│ 前导码   │ 同步字  │ 头部 (4 字节)     │ 载荷 (N 字节)│ CRC-16   │ 结束标记 │
│ (64 bit) │(16 bit)│ seq|flags|rsv    │            │ (16 bit) │ (16 bit) │
└──────────┴────────┴──────────────────┴────────────┴──────────┘
```

- **前导码**：64 位交替比特序列 [1,0,1,0,...]，用于时钟恢复和自动增益控制
- **同步字**：`0x1A2B`，帧起始标记
- **头部**：序列号 (2B) + 标志 (1B) + 保留 (1B)
- **载荷**：可变长度，默认 512 字节
- **CRC-16**：CCITT 多项式 `0x1021`
- **结束标记**：`0xBEEF`

### 四阶段双向协议

```
阶段 1: 握手
  Sender   ─── HELLO (文件名, 大小, SHA-256, ...) ───→  Receiver
  Sender   ←── ACK (就绪确认) ────────────────────  Receiver

阶段 2: 批量传输
  Sender   ─── PKT[0] ───→
  Sender   ─── PKT[1] ───→
           ...               Receiver 持续收集并缓冲
  Sender   ─── PKT[N-1] ──→ (标记 FLAG_LAST)

阶段 3: 选择性重传
  Sender   ←── NACK (缺失包位掩码) ──────────────  Receiver
  Sender   ─── 仅重发缺失的包 ──────────────────→  Receiver

阶段 4: 完成确认
  Sender   ←── DONE (SHA-256 校验通过) ─────────  Receiver
```

## 命令行参数

### `send.py` — 发送端

```
用法: send.py [选项] <文件>

选项:
  --mode {2fsk,4fsk}    调制方案 (默认: 4fsk)
  --baud {100,200,300,400,600}
                        波特率（符号/秒），默认: 300
  --packet-size N        每包载荷字节数（默认: 512）
  --device DEVICE       音频输出设备名称或索引
  --list-devices        列出设备并退出
  --test-tone           播放测试音以验证输出
```

### `receive.py` — 接收端

```
用法: receive.py [选项] <输出文件>

选项:
  --mode {2fsk,4fsk}    调制方案（需与发送端一致）
  --baud {100,200,300,400,600}
                        波特率（需与发送端一致）
  --timeout SECONDS     最长监听时间（默认: 120 秒）
  --device DEVICE       音频输入设备名称或索引
  --list-devices        列出设备并退出
```

## 性能参数

### 理论速率

| 调制方式 | 波特率 | 每符号比特 | 原始速率 | 净速率（约） | 1 MB 传输时间 |
|----------|--------|------------|----------|-------------|---------------|
| 2-FSK | 100 | 1 | 100 bps | ~85 bps | ~95 秒 |
| 2-FSK | 300 | 1 | 300 bps | ~255 bps | ~32 秒 |
| 4-FSK | 100 | 2 | 200 bps | ~170 bps | ~48 秒 |
| **4-FSK** | **300** | **2** | **600 bps** | **~500 bps** | **~17 秒** |
| 4-FSK | 600 | 2 | 1200 bps | ~1000 bps | ~8 秒 |

> 净速率已扣除 RS 编码冗余 (~6.7%)、协议头部、CRC 和帧同步开销。

### 物理层参数

| 参数 | 值 |
|------|----|
| 采样率 | 44100 Hz（CD 音质） |
| 4-FSK 频率 | 2000 / 2667 / 3333 / 4000 Hz |
| 调制方式 | 相位连续 FSK + 升余弦包络 |
| 解调方式 | 非相干正交匹配滤波 |
| 同步方式 | 归一化互相关（粗同步）+ 比特匹配微调（精同步） |
| 最大文件大小 | 4 MB |

### Reed-Solomon 纠错

| 参数 | 值 |
|------|----|
| 码字长度 (n) | 255 字节 |
| 数据长度 (k) | 239 字节 |
| 校验冗余 | 16 字节 (~6.7%) |
| 纠错能力 | 每 255 字节块最多纠正 **8 个字节错误** |
| 实现方式 | 纯 Python GF(2^8) + Berlekamp-Massey + Chien搜索 + Forney算法 |

## 研究背景

本项目源于一个研究问题：**能否仅依靠物理音频端口在两台计算机之间可靠地传输任意文件？**

### 设计决策

1. **为什么选 FSK？**  
   FSK 抗噪声能力强，适合音频信道的非线性失真和带限特性。相比 ASK/PSK，它天然适应音频线的不平坦频率响应。

2. **为什么选 4-FSK？**  
   在 2000-4000 Hz（电话语音带宽）内可容纳 4 个 667 Hz 间隔的载波，每个符号携带 2 比特，吞吐量翻倍且硬件要求不变。

3. **为什么用 Reed-Solomon？**  
   音频信道的主要误码特征是突发错误（瞬时噪声会导致连续多个符号误判），RS 码对突发错误有天然优势。RS(255,239) 可纠正每 255 字节中最多 8 字节错误，开销仅 6.7%。

4. **为什么不用 OFDM 或卷积码？**  
   本项目目标是在 "仅 Python + 标准音频 I/O" 的约束下探索传输极限，更复杂的方案会显著增加实现复杂度和依赖。
